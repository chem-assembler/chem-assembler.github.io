# スマホ対応（レスポンシブ・縦横両対応）設計書（P11）

スマホで**キャンバス（作図領域）を最優先に確保**しつつ、道具・お題パネルをドロワー化して
限られた画面を有効に使う。縦横どちらの向きでも使えるようにする。
2026-07-24 ユーザー合意（縦横両対応・設計書先行）に基づく。実装前の合意用ドキュメント。

---

## 1. 課題と方針

- 現状は3カラム固定（左ツール280px・中央キャンバス・右パネル320px）。狭い画面ではキャンバスがほぼ潰れる。
- 方針: **キャンバスは常に最大化**し、ツールと右パネルは「必要なときに開くドロワー/シート」にする。
- **PC（広い画面）の見た目・挙動は一切変えない**。メディアクエリで狭い画面のみ別レイアウトにする。
- 座標系は `getScreenCTM()` 依存なので、パネルサイズが変わっても作図の座標は自動追従する（設計上の強み。ここは触らない）。
- 先日のモード再編（3モードで右パネルの中身を絞る仕組み）がスマホでそのまま効く。

## 2. ブレークポイント

| 区分 | 条件 | レイアウト |
|---|---|---|
| デスクトップ | `min-width: 900px` | 現行の3カラム（変更なし） |
| モバイル縦 | `max-width: 899px` かつ `orientation: portrait` | キャンバス上・ツール下バー・右パネルは下シート |
| モバイル横 | `max-width: 899px` かつ `orientation: landscape` | 細い左ツール列・中央キャンバス・右はドロワー |

- 判定は幅と向きの組み合わせ。タブレット（≥900px）は当面デスクトップ扱い。

## 3. モバイル縦レイアウト

```
┌──────────────────────────┐
│ ヘッダー（ロゴ・モードタブ・❓） │ ← 細い固定バー
├──────────────────────────┤
│                          │
│       キャンバス          │ ← 画面の大半（flex:1）
│   （全体表示・ズーム・パン）  │
│                          │
├──────────────────────────┤
│ [お題/反応 ▲]  ← 右パネルをシートで開くボタン
├──────────────────────────┤
│ 🛠ツールバー（横スクロール）  │ ← 操作モード・結合・原子・モジュール
└──────────────────────────┘
```

- **ヘッダー**: ロゴは小さく、モードタブ（🧩📚🧪）を主役に。❓ヘルプはアイコンのみ。
- **キャンバス**: 縦画面の中央を占有。上部に座標表示・全体表示・全消去などのミニツールバー（既存 canvas-header を縮小）。
- **ツールバー（下）**: 親指の届く最下部。操作モード（Select/Bond/Erase/✳）・結合タイプ・原子パレット・モジュールを**横スクロールの帯**にまとめる。ボタンは指で押しやすいサイズ（最小44px）。
- **右パネル（お題・反応など）**: 「お題/反応 ▲」ボタンで**下からせり上がるシート**として開く。閉じるとキャンバスが最大。モードで中身が変わる（既存の data-modes をそのまま利用）。

## 4. モバイル横レイアウト

```
┌──┬────────────────────┬──┐
│ツ │                    │パ│
│ー │     キャンバス      │ネ│  ← 左右は細い列（アイコン中心）
│ル │                    │ル│
└──┴────────────────────┴──┘
```

- 横は幅に余裕があるので、**現行の3カラムを圧縮した形**に近づける。
- 左ツールはアイコン主体の細い列、右パネルは折りたたみ可能な細い列（またはドロワー）。
- モードタブは上部の細いバー。

## 5. 実装方針（CSSと最小限のJS）

- **CSS中心**: 既存のDOM構造を保ち、`@media` で `main` の `flex-direction`・パネル幅・順序（`order`）を切り替える。要素の移動はCSSの再配置で行い、DOMは極力動かさない。
- **ドロワー/シート**: CSS `transform: translateY()` ＋ トグルボタンでスライド。フレームワーク不使用（開発方針2章）。開閉状態は body のクラスかトグルで管理。
- **横スクロールのツール帯**: `overflow-x:auto; display:flex;` で実現。
- **タッチ判定**: 既にPointer Events統一・`touch-action:none`・ピンチズーム対応済み（P7-7）。ボタンのタップ領域だけ拡大する。
- **viewport**: 既に `width=device-width, initial-scale=1` 設定済み。`100dvh`（動的ビューポート高さ）を使い、モバイルブラウザのアドレスバー増減でキャンバス高さが破綻しないようにする。
- **モーダル類**（クイズ・異性体ギャラリー・立体・チュートリアルFAQ）: `max-height:90dvh; overflow:auto;` を徹底し、小画面で溢れないようにする。

## 6. マイルストーン

- **M1**: ブレークポイント導入＋モバイル縦レイアウト（キャンバス最大化・下ツールバー・右パネルの下シート化）。まず縦で「使える」状態に。
- **M2**: モバイル横レイアウト（圧縮3カラム）＋向き切替時の破綻防止。
- **M3**: 仕上げ（タップ領域拡大・ドロワーのアニメ・モーダルの小画面最適化・チュートリアルのゴーストカーソルが小画面でも正しく動くか確認）。

## 7. リスク・未決事項

- **回帰テスト**: iframe は広いビューポートで動くためメディアクエリは発火せず、既存テストは影響を受けない見込み。念のためモバイル幅での要素可視性テスト（M1後）を1本追加する。
- **ドロワーと反応機構モード**: 反応機構ビューアは右パネルにあるため、シート内での操作性を確認する。
- **チュートリアルのデモ**: ゴーストカーソルはクライアント座標で動くので、パネル移動後も座標変換は追従するはず。ただしボタンの `getBoundingClientRect` がドロワー閉状態だと0になる点に注意（デモ再生時はドロワーを開く/該当モードにする処理を検討）。
- **PCへの非干渉**: `min-width:900px` 側は現状維持。モバイルCSSが漏れてPC表示を壊さないよう、メディアクエリ内に閉じる。
- **既存のインラインstyle**: 右パネル要素にインラインstyleが多く、CSSで上書きしづらい箇所は `!important` かクラス化で対応（将来的にインラインstyleをCSSへ寄せる。開発方針2章の方向）。

## 8. 段階的リリースの安全策

- モバイルレイアウトはメディアクエリで隔離するため、PCユーザーには影響しない。M1を入れても既存の遊び方は保たれる。
- 各Mの後、実機（ユーザー）で縦・横を確認して次へ進む（P7-7のタッチ確認と同じ運用）。
